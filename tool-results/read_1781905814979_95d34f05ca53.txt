    73→## 2. Contexto e Escopo
    74→
    75→### Problem Statement
    76→
    77→O módulo financeiro do Sistema de Gestão Saúde & Proteção (SPF) enfrenta três problemas críticos que impedem a escalabilidade do modelo de negócio:
    78→
    79→**1. Fragilidade Financeira e Risco de Fraude:** O sistema atual não possui idempotência robusta em transações de bonificação multinível, resultando em duplicidade de pagamentos em cenários de retry de rede ou falhas parciais. Além disso, a ausência de controles de saldo devedor permite que revendedores realizem saques ou abatimentos mesmo após cancelamentos de contratos que deveriam estornar bonificações já liberadas.
    80→
    81→**2. Inconsistência Tributária e Compliance:** A mistura de receita tributável (clube de benefícios) com repasse de risco (prêmio de seguro) em uma única transação financeira gera exposição fiscal (ISS indevido sobre prêmios) e viola diretrizes SUSEP. O sistema não possui separação contábil clara, nem retenção automática de IRRF/INSS em saques de Pessoa Física, criando passivos trabalhistas e tributários.
    82→
    83→**3. Opacidade Operacional:** A falta de reconciliação automática entre boletos pagos, bonificações liberadas e abatimentos via carteira digital impede auditoria financeira confiável. O time financeiro não possui visibilidade em tempo real de saldos devedores, transações pendentes de aprovação manual ou conflitos de rede (CPF ativo em múltiplas árvores de patrocínio).
    84→

   145→### Glossário
   146→
   147→| Termo | Definição | Exemplo |
   148→|-------|-----------|---------|
   149→| **Bonificação** | Remuneração paga a revendedores por indicação/fechamento de novos clientes. **NUNCA** usar "comissão" ou "comissionamento" para evitar caracterização de corretagem (SUSEP) | R$ 50,00 por novo contrato |
   150→| **Carteira Digital** | Conta virtual do revendedor com saldos: disponível, bloqueado, devedor | Saldo Disponível: R$ 500,00 |
   151→| **Estorno** | Reversão de bonificação já liberada, debitando do saldo disponível (pode gerar saldo devedor) | Contrato cancelado → estorna R$ 50,00 |
   152→| **Abatimento** | Uso de saldo disponível para quitar parcelas de contratos próprios ou de terceiros | Revendedor paga parcela de R$ 100,00 com saldo |
   153→| **Saldo Devedor** | Valor negativo na carteira após estorno que excedeu saldo disponível | Saldo: -R$ 100,00 (bloqueia novos saques) |
   154→| **Idempotência** | Garantia de que operação executada múltiplas vezes produz mesmo resultado | Constraint UNIQUE impede bonificação duplicada |
   155→| **Maker/Checker** | Fluxo de aprovação manual para operações acima de limite configurado | Saque > R$ 1.000,00 exige aprovação do Financeiro |
   156→| **RPA (Recibo de Pagamento Autônomo)** | Documento fiscal para pagamentos a Pessoa Física sem vínculo empregatício | Saque PF com retenção IRRF/INSS |
   157→| **Air-Gap Financeiro** | Separação contábil estrita entre receita tributável (clube) e repasse de risco (prêmio) | NF emitida apenas sobre serviço, não sobre prêmio |

   330→## 4. Especificação Funcional Detalhada
   331→
   332→### 4.1 Regras de Negócio (Formato SE-ENTÃO-SENÃO)
   333→
   334→#### RN-001: Base de Cálculo da Bonificação
   335→
   336→**SE** contrato for aprovado  
   337→**ENTÃO** calcular bonificação aplicando `percentual_indicacao` ou `percentual_fechamento` (conforme nível na árvore) sobre:  
   338→- Taxa de Adesão (se pagamento único)  
   339→- Valor da Parcela Mensal/Anual  
   340→**SENÃO** (se tentar incluir prêmio de seguro ou capital segurado)  
   341→**EXCEÇÃO** Bloquear operação com erro `BONUS_BASE_CALCULO_INVALIDO` e logar auditoria
   342→
   343→**Justificativa:** Compliance SUSEP - bonificação remunera atividade comercial, não intermediação de seguro
   344→
   345→---
   346→
   347→#### RN-002: Idempotência de Bonificação
   348→
   349→**SE** sistema tentar criar `transacoes_bonificacao` com mesma combinação `(origem_contrato_id, carteira_id, nivel_origem)`  
   350→**ENTÃO** PostgreSQL constraint UNIQUE impedirá inserção (violates unique constraint)  
   351→**SENÃO** (se aplicação tentar retry sem verificar constraint)  
   352→**EXCEÇÃO** Retornar `409 Conflict` com mensagem "Bonificação já processada para este contrato e nível"
   353→
   354→**Implementação:**
   355→```sql
   356→CREATE UNIQUE INDEX idx_bonificacao_unica
   357→ON transacoes_bonificacao (origem_contrato_id, carteira_id, nivel_origem);
   358→```
   359→
   360→---
   361→
   362→#### RN-003: Abatimento de Parcela com Saldo Insuficiente
   363→
   364→**SE** revendedor tentar abater parcela  
   365→**E** `saldo_disponivel` <= 0  
   366→**ENTÃO** bloquear operação e retornar erro `SALDO_INSUFICIENTE`  
   367→**SENÃO** (se `saldo_disponivel` > 0 mas < `valor_parcela`)  
   368→**EXCEÇÃO** Permitir abatimento parcial, gerar boleto residual com `valor_restante`
   369→
   370→**Cálculo:**
   371→```typescript
   372→const valorAbatido = Math.min(saldoDisponivel, valorParcela);
   373→const saldoRestante = valorParcela - valorAbatido;
   374→const statusConta = saldoRestante === 0 ? 'PAGO' : 'PARCIALMENTE_PAGO';
   375→```
   376→
   377→---
   378→
   379→#### RN-004: Estorno Pós-Aprovação (Cancelamento de Contrato)
   380→
   381→**SE** contrato com status `APROVADO` for cancelado  
   382→**ENTÃO** para cada `transacoes_bonificacao` com status `LIBERADO`:  
   383→1. Alterar status para `ESTORNADO`  
   384→2. Debitar `valor` de `saldo_disponivel` da carteira  
   385→3. **SE** `saldo_disponivel` < 0 após débito:  
   386→   - Transferir valor absoluto para `saldo_devedor`  
   387→   - Zerar `saldo_disponivel`  
   388→**SENÃO** (se bonificação já foi usada em `transacoes_pagamento`)  
   389→**EXCEÇÃO** Manter débito negativo (saldo devedor), **NÃO** reverter pagamento da parcela (boleto já foi emitido/pago)
   390→
   391→---
   392→
   393→#### RN-005: Saque de Bonificação com Retenção Tributária (PF)
   394→
   395→**SE** revendedor for Pessoa Física  
   396→**E** `configuracoes_regras_negocio.SAQUE_PF_ATIVO = true`  
   397→**E** `valor_saque` > 0  
   398→**ENTÃO**:  
   399→1. Calcular IRRF conforme tabela progressiva 2026  
   400→2. Calcular INSS (11% até teto R$ 908,85 em 2026)  
   401→3. Persistir `valor_irrf_retido` e `valor_inss_retido`  
   402→4. Creditar apenas `valor_liquido = valor_bruto - irrf - inss`  
   403→5. Gerar DARF/GPS no módulo Financeiro  
   404→**SENÃO** (se `SAQUE_PF_ATIVO = false`)  
   405→**EXCEÇÃO** Bloquear saque com erro `SAQUE_PF_DESATIVADO`
   406→
   407→---
   408→
   409→#### RN-006: Maker/Checker para Saques Acima do Limite
   410→
   411→**SE** `valor_saque` > `configuracoes_regras_negocio.LIMITE_SAQUE_DIARIO`  
   412→**ENTÃO**:  
   413→1. Criar `transacoes_pagamento` com status `PENDENTE_APROVACAO`  
   414→2. Notificar Financeiro via dashboard e email  
   415→3. Bloquear crédito até aprovação manual  
   416→**SENÃO** (se `valor_saque` <= limite)  
   417→**EXCEÇÃO** Processar imediatamente (status `CONCLUIDO`)
   418→
   419→---
   420→
   421→#### RN-007: Bloqueio de Saque com Saldo Devedor
   422→
   423→**SE** revendedor possuir `saldo_devedor` > 0  
   424→**ENTÃO**:  
   425→1. Bloquear novos saques  
   426→2. Priorizar abatimento automático de futuras bonificações para cobrir saldo devedor antes de liberar `saldo_disponivel`  
   427→**SENÃO** (se revendedor tentar saque mesmo com saldo devedor)  
   428→**EXCEÇÃO** Retornar erro `SALDO_DEVEDOR_EXISTENTE` com mensagem: "Regularize seu saldo devedor de R$ X antes de realizar novos saques"
   429→
   430→---
   431→
   432→#### RN-008: Recorrência Automática (Geração de Parcelas)
   433→
   434→**SE** job diário executar às 02:00 AM  
   435→**E** contrato.status = `APROVADO`  
   436→**E** contrato.periodicidade = `MENSAL`  
   437→**E** última parcela quitada/vencida há > 5 dias  
   438→**ENTÃO**:  
   439→1. Recalcular valor baseado em agregados ativos no momento  
   440→2. Criar nova `contas_a_pagar` com `data_vencimento` = próximo ciclo  
   441→3. Gerar boleto e atualizar `contratos_legado_flat` (apenas campos financeiros)  
   442→**SENÃO** (se contrato estiver `SUSPENSO`)  
   443→**EXCEÇÃO** Não gerar nova parcela até regularização
   444→

   447→#### RN-009: Anti-Conflito de Rede (Cross-Tree)
   448→
   449→**SE** tentativa de cadastro de novo revendedor  
   450→**E** CPF já existe em outra árvore de patrocínio ativa (`data_fim_vinculo IS NULL`)  
   451→**ENTÃO** bloquear com erro `CONFLITO_DE_REDE`  
   452→**SENÃO** (se CPF existe mas com `data_fim_vinculo` preenchido)  
   453→**EXCEÇÃO** Permitir cadastro (revendedor inativo pode ser reativado em nova árvore)
   454→

   456→
   457→#### RN-010: Air-Gap de Auto-Patrocínio
   458→
   459→**SE** tentativa de criar vínculo de patrocínio  
   460→**E** `revendedor_id` = `patrocinador_id`  
   461→**OU** `revendedor_id` é dependente direto (cônjuge/filho) de `patrocinador_id`  
   462→**ENTÃO** bloquear com erro `AUTO_PATROCINIO_PROIBIDO`  
   463→**SENÃO** (se for parentesco indireto: irmão, tio, primo)  
   464→**EXCEÇÃO** Permitir (mas logar auditoria para revisão manual)

   718→### 4.4 Edge Cases e Exceções (30% da Seção)
   719→
   720→#### 4.4.1 Cenários de Concorrência e Race Conditions
   721→
   722→##### EC-001: Múltiplos Abatimentos Simultâneos na Mesma Carteira
   723→
   724→**Cenário:** Revendedor abre duas abas do navegador e clica "Abater Parcela" simultaneamente em duas parcelas diferentes, ambas com valor R$ 300,00. Saldo disponível é R$ 500,00.
   725→
   726→**Comportamento Esperado:**
   727→1. Primeira requisição adquire lock distribuído no Redis (`SET carteira:{id}:lock NX PX 5000`)
   728→2. Primeira requisição inicia transação PostgreSQL (`BEGIN`)
   729→3. Primeira requisição consulta saldo com `SELECT saldo_disponivel FROM carteiras_digitais WHERE id = ? FOR UPDATE` (lock de linha)
   730→4. Segunda requisição tenta adquirir lock no Redis e falha (timeout 5s)
   731→5. Segunda requisição retorna `409 Conflict` com mensagem: "Operação em andamento. Tente novamente em alguns segundos."
   732→6. Primeira requisição processa abatimento, commit, libera lock
   733→
   734→**Implementação:**
   735→```typescript
   736→async function abaterParcela(carteiraId: string, contaId: string, valor: number) {
   737→  const lockKey = `carteira:${carteiraId}:lock`;
   738→  const lockAcquired = await redis.set(lockKey, '1', 'EX', 5, 'NX');
   739→  
   740→  if (!lockAcquired) {
   741→    throw new ConflictException('Operação em andamento. Tente novamente.');
   742→  }
   743→  
   744→  try {
   745→    return await prisma.$transaction(async (tx) => {
   746→      const carteira = await tx.carteirasDigitais.findUnique({
   747→        where: { id: carteiraId },
   748→        lock: 'UPDATE' // SELECT ... FOR UPDATE
   749→      });
   750→      
   751→      if (carteira.saldoDisponivel < valor) {
   752→        throw new BadRequestException('Saldo insuficiente');
   753→      }
   754→      
   755→      // Processar abatimento...
   756→      
   757→      return { success: true };
   758→    });
   759→  } finally {
   760→    await redis.del(lockKey);
   761→  }
   762→}
   763→```
   764→
   765→##### EC-002: Aprovação de Contrato e Cancelamento Simultâneos
   766→
   767→**Cenário:** SuperAdmin clica "Aprovar" no contrato C500. No mesmo milissegundo, job de cancelamento por inadimplência tenta cancelar o mesmo contrato.
   768→
   769→**Comportamento Esperado:**
   770→1. Aprovação adquire lock de linha em `contratos` (`SELECT ... FOR UPDATE`)
   771→2. Aprovação inicia transação e altera status para `APROVADO`
   772→3. Job de cancelamento tenta adquirir lock e bloqueia (wait)
   773→4. Aprovação commit e libera lock
   774→5. Job de cancelamento adquire lock, verifica status = `APROVADO`, e **aborta** (não cancela contrato recém-aprovado)
   775→6. Job loga: "Contrato C500 ignorado (status alterado durante processamento)"
   776→
   777→**Constraint de Banco:**
   778→```sql
   779→-- Trigger para impedir cancelamento de contrato aprovado há menos de 1 minuto
   780→CREATE OR REPLACE FUNCTION impedir_cancelamento_imediato()
   781→RETURNS TRIGGER AS $$
   782→BEGIN
   783→  IF NEW.status = 'CANCELADO' 
   784→     AND OLD.status = 'APROVADO'
   785→     AND NOW() - OLD.updated_at < INTERVAL '1 minute'
   786→  THEN
   787→    RAISE EXCEPTION 'Contrato recém-aprovado não pode ser cancelado imediatamente';
   788→  END IF;
   789→  RETURN NEW;
   790→END;
   791→$$ LANGUAGE plpgsql;
   792→
   793→CREATE TRIGGER trigger_impedir_cancelamento
   794→BEFORE UPDATE ON contratos
   795→FOR EACH ROW
   796→EXECUTE FUNCTION impedir_cancelamento_imediato();
   797→```

   799→##### EC-003: Webhook de Pagamento Duplicado (Gateway Retry)
   800→
   801→**Cenário:** Gateway de pagamento envia webhook de confirmação de pagamento do boleto B100. Rede falha, gateway faz retry 3 vezes com mesmo payload.
   802→
   803→**Comportamento Esperado:**
   804→1. Primeiro webhook recebido: sistema cria `contas_a_pagar` com status `PAGO`, gera bonificação, etc.
   805→2. Segundo webhook recebido: sistema verifica idempotency key (campo `gateway_transaction_id` na tabela `webhooks_recebidos`)
   806→3. Sistema detecta duplicação e retorna `200 OK` (sem processar novamente)
   807→4. Sistema loga: "Webhook duplicado ignorado (transaction_id: X)"
   808→
   809→**Implementação:**
   810→```typescript
   811→// Middleware de idempotência
   812→async function idempotencyMiddleware(req, res, next) {
   813→  const idempotencyKey = req.headers['x-idempotency-key'] || req.body.transaction_id;
   814→  
   815→  if (!idempotencyKey) {
   816→    return next();
   817→  }
   818→  
   819→  const existing = await prisma.webhooksRecebidos.findUnique({
   820→    where: { transactionId: idempotencyKey }
   821→  });
   822→  
   823→  if (existing) {
   824→    return res.status(200).json({ message: 'Already processed', duplicate: true });
   825→  }
   826→  
   827→  await prisma.webhooksRecebidos.create({
   828→    data: {
   829→      transactionId: idempotencyKey,
   830→      payload: req.body,
   831→      processedAt: new Date()
   832→    }
   833→  });
   834→  
   835→  next();
   836→}

   843→##### EC-004: Bonificação para Revendedor sem Carteira Criada
   844→
   845→**Cenário:** Revendedor antigo (cadastrado antes da implementação de carteiras digitais) indica novo cliente. Sistema tenta criar `transacoes_bonificacao` mas `carteiras_digitais` não existe para este `titular_id`.

   855→async function garantirCarteira(titularId: string): Promise<CarteiraDigital> {

   860→  if (!carteira) {
   861→    carteira = await prisma.carteirasDigitais.create({
   862→      data: {
   863→        titularId,
   864→        saldoDisponivel: 0,
   865→        saldoBloqueado: 0,
   866→        saldoDevedor: 0
   867→      }
   868→    });
   869→    logger.info(`Carteira criada automaticamente para revendedor ${titularId}`);
   870→  }
   871→  
   872→  return carteira;
   873→}

   889→#### 4.4.3 Cenários de Timeout e Falhas de Dependências Externas
   890→
   891→##### EC-006: Timeout na Geração de Boleto (Gateway Indisponível)
   892→
   893→**Cenário:** Sistema tenta gerar boleto via API do gateway, mas gateway retorna timeout após 30s.
   894→
   895→**Comportamento Esperado:**
   896→1. Sistema tenta 3 vezes com backoff exponencial (1s, 2s, 4s)
   897→2. Se todas falharem:
   898→   - Sistema cria `contas_a_pagar` com status `PENDENTE` mas sem `boleto_url`
   899→   - Sistema publica evento `BoletoGeracaoFalhou` na fila
   900→   - Job de retry tenta novamente em 1h, 6h, 24h
   901→   - Sistema envia email para titular: "Estamos com instabilidade na geração do seu boleto. Tentaremos novamente em breve."
   902→   - Sistema notifica DevOps via PagerDuty
   903→3. Frontend exibe: "Boleto em processamento. Você receberá por email em até 24h."
   904→
   905→**Implementação:**
   906→```typescript
   907→const retryPolicy = {
   908→  maxAttempts: 3,
   909→  backoff: {
   910→    type: 'exponential',
   911→    delay: 1000 // 1s, 2s, 4s
   912→  },
   913→  retryableErrors: ['TIMEOUT', 'NETWORK_ERROR', '503']
   914→};
   915→
   916→async function gerarBoleto(contaId: string): Promise<string> {
   917→  try {
   918→    return await retryWithBackoff(
   919→      () => paymentGateway.createBoleto(contaId),
   920→      retryPolicy
   921→    );
   922→  } catch (error) {
   923→    await prisma.contasAPagar.update({
   924→      where: { id: contaId },
   925→      data: { status: 'PENDENTE', boletoUrl: null }
   926→    });
   927→    
   928→    await messageBroker.publish('BoletoGeracaoFalhou', { contaId, error: error.message });
   929→    await notificationService.sendEmail(titularId, 'boleto_pendente');
   930→    
   931→    throw new ServiceUnavailableException('Falha ao gerar boleto');
   932→  }
   933→}
   934→```
   935→
   936→##### EC-007: Redis Indisponível (Cache de Saldos)
   937→
   938→**Cenário:** Redis cluster cai durante pico de acesso ao dashboard de revendedores.
   939→
   940→**Comportamento Esperado:**
   941→1. Sistema detecta falha de conexão com Redis (timeout 100ms)
   942→2. Sistema ativa modo "fallback para DB"
   943→3. Todas as consultas de saldo vão direto para PostgreSQL (mais lento, mas funcional)
   944→4. Sistema loga warning: "Redis indisponível, usando fallback para DB"
   945→5. Sistema envia alerta para DevOps (PagerDuty nível 2)
   946→6. Quando Redis voltar, sistema repopula cache gradualmente (cache warming)
   947→
   948→**Implementação:**
   949→```typescript
   950→class SaldoService {
   951→  async getSaldoDisponivel(carteiraId: string): Promise<number> {
   952→    try {
   953→      const cached = await redis.get(`carteira:${carteiraId}:saldo`);
   954→      if (cached) return parseFloat(cached);
   955→    } catch (error) {
   956→      logger.warn('Redis indisponível, usando fallback para DB');
   957→    }
   958→    
   959→    // Fallback para DB
   960→    const carteira = await prisma.carteirasDigitais.findUnique({
   961→      where: { id: carteiraId }
   962→    });
   963→    
   964→    return carteira.saldoDisponivel;
   965→  }
   966→}
   967→```

   973→##### EC-008: Valor de Abatimento Negativo ou Zero
   974→
   975→**Cenário:** Frontend envia `valor_abatido = -100` ou `0` devido a bug de manipulação de input.
   976→
   977→**Comportamento Esperado:**
   978→1. Backend valida: `if (valor <= 0) throw new BadRequestException('Valor deve ser positivo')`

   980→3. Loga auditoria: "Tentativa de abatimento com valor inválido: {valor}"
   981→
   982→##### EC-009: Conta a Pagar já Cancelada
   983→
   984→**Cenário:** Revendedor tenta abater parcela que já foi cancelada (contrato cancelado).
   985→
   986→**Comportamento Esperado:**
   987→1. Sistema consulta `contas_a_pagar` e verifica `status = 'CANCELADO'`
   988→2. Retorna `409 Conflict`: "Esta parcela foi cancelada e não pode ser abatida"
   989→3. Não cria transação
   990→
   991→##### EC-010: Saque para Conta Bancária Inválida
   992→
   993→**Cenário:** Revendedor cadastra conta bancária com agência/dígito incorretos. Gateway rejeita transferência.
   994→
   995→**Comportamento Esperado:**
   996→1. Gateway retorna erro `INVALID_BANK_ACCOUNT`
   997→2. Sistema altera `transacoes_pagamento.status` para `ESTORNADO`
   998→3. Sistema credita valor de volta em `saldo_disponivel`
   999→4. Sistema envia email: "Seu saque de R$ X foi estornado. Motivo: conta bancária inválida. Atualize seus dados e tente novamente."
  1000→5. Sistema loga auditoria
