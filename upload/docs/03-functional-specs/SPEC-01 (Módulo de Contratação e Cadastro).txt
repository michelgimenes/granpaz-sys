## 2. Contexto e Escopo

### Problem Statement
O fluxo de contratação de planos de proteção familiar envolve uma hierarquia complexa de vínculos (Titular, Dependentes, Agregados, Sub-dependentes) e regras de elegibilidade rigorosas (idade, estado civil, parentesco). A ausência de uma camada de cadastro unificada e agnóstica resulta em duplicidade de código, validações client-side contornáveis e violações de compliance SUSEP (ex: exposição de dados de seguradora ao cliente final).

É imperativo estabelecer um componente de entrada único (`PessoaFisicaForm`) com injeção de contexto estrita e validação determinística no backend para garantir a integridade referencial e a blindagem jurídica do ecosistema. Adicionalmente, a falta de alinhamento entre os schemas de cadastro e os módulos financeiros downstream gerava inconsistências na persistência de retenções tributárias e idempotência de pagamentos, problema este resolvido nesta versão através da harmonização forçada das tabelas financeiras globais.

### Objetivos
| Categoria | Métrica / KPI | Alvo |
| :--- | :--- | :--- |
| **Performance** | Latência p99 da API de submissão de contrato (`POST /contratos`) | `< 350ms` |
| **Performance** | Tempo de resposta da busca por CPF (`GET /buscar-por-cpf`) | `< 100ms` |
| **Qualidade** | Cobertura de Testes E2E (Cypress/Playwright) nos fluxos de contratação | `100%` |
| **Segurança** | Taxa de injeção de `tipo_registro` inválido via manipulação de payload | `0%` (Bloqueio 400) |
| **Negócio** | Taxa de conversão de submissão sem erros de validação bloqueante | `> 85%` |
| **Integridade** | Divergência de Schema entre Módulo 1 e Módulo 3 | `0` (Zero Tolerance) |

### Out of Scope (Não será implementado nesta versão)
1.  **Módulo de Aprovação (Backoffice):** Fluxo de análise manual pelo SuperAdmin (Vide SPEC-02).
2.  **Gestão de Sinistros:** Acionamento de cobertura e validação de carência (Vide SPEC-04).
3.  **Processamento Financeiro Efetivo:** Geração de boletos, saques e cálculo de bonificações (Vide SPEC-03). *Nota: Este módulo apenas persiste as estruturas de dados necessárias para o Módulo 3.*
4.  **Módulo de Rede Multinível:** Visualização e gestão da árvore de patrocínios.
5.  **Integração Direta com Seguradora:** O vínculo jurídico é postergado para aprovação manual.

### Premissas e Restrições
-   **Stack Tecnológica:** Next.js (Frontend BFF), Node.js/TypeScript (API Core), PostgreSQL 15+ (Persistência).
-   **Compliance:** LGPD (Anonimização futura), SUSEP (Air-Gap de Seguradora), CDC (Arrependimento 7 dias).
-   **Restrição de Banco:** Utilização estrita de `CHECK CONSTRAINTS` e `PARTIAL UNIQUE INDEXES` no PostgreSQL para garantir integridade no nível de dados.
-   **Verdade Absoluta:** Os schemas definidos na Seção 2.6 deste documento são mandatórios e sobrescrevem quaisquer definições anteriores em outros SPECs.

### Glossário
| Termo | Definição |
| :--- | :--- |
| **Air-Gap** | Isolamento lógico/funcional de dados sensíveis ou críticos (ex: Seguradora) de interfaces públicas. |
| **Estipulante** | Pessoa jurídica (Saúde & Proteção) que contrata seguro coletivo em nome de seus clientes, sem assumir o risco. |
| **Idade Transiente** | Validação de idade aplicada apenas no ato do cadastro inicial, desativada em edições subsequentes (direito adquirido). |
| **Récita** | Fluxo de reativação de contrato suspenso por inadimplência, com recálculo de carência. |
| **Sub-dependente** | Dependente vinculado a um Agregado (e não diretamente ao Titular). |
| **Saldo Devedor** | Passivo financeiro gerado quando estornos excedem o saldo disponível; campo obrigatório na tabela de carteiras. |

### 3.3 Architecture Decision Records (ADRs)
| ID | Decisão | Alternativas Consideradas | Justificativa | Consequências |
| :--- | :--- | :--- | :--- | :--- |
| **ADR-001** | **Componentização Contextual Estrita** (`PessoaFisicaForm` agnóstico). | Componentes separados por tipo (`TitularForm`, `DependenteForm`). | Reduz drasticamente a dívida técnica de UI. O tipo é injetado via props. | Backend deve validar rigorosamente o `tipo_registro` vs Rota. |
| **ADR-002** | **Air-Gap de Seguradora no Frontend Público**. | Exibir lista de seguradoras e deixar o cliente escolher. | Compliance SUSEP. O sistema atua como Estipulante; aceitação é exclusiva do Backoffice. | Frontend nunca carrega `seguradoras`. API retorna 403 para não-Admins. |
| **ADR-003** | **CPF VARCHAR(14) Futuro-Proof**. | BIGINT ou VARCHAR(11). | Preparação para futura inclusão de letras no CPF brasileiro. | Regex de validação no frontend deve ser atualizável, DB não exige migração. |
| **ADR-004** | **Obsolescência da Tabela Flat (`contratos_legado_flat`)**. | Manter sincronização bidirecional completa. | A flat é legada para boletos. Vínculos residem EXCLUSIVAMENTE nas tabelas normalizadas. | Relatórios e PDFs devem consumir API normalizada (JOINs). |
| **ADR-005** | **Harmonização Antecipada de Schema Financeiro**. | Definir schema financeiro apenas no SPEC-03. | Evita retrabalho de migração e garante que o Módulo 1 já crie as estruturas corretas para o Módulo 3. | Tabelas `carteiras_digitais`, `transacoes_pagamento` e `webhooks_recebidos` são criadas neste módulo. |

## 4. Especificação Funcional Detalhada

### 4.1 Regras de Negócio (Determinísticas)
| ID | Regra de Negócio (SE / ENTÃO / SENÃO) |
| :--- | :--- |
| **RN-01** | **SE** o usuário submeter um novo contrato **ENTÃO** o sistema deve verificar se já existe contrato com status `IN ('AGUARDANDO_APROVACAO', 'APROVADO', 'SUSPENSO')` para o mesmo `titular_id`. **SENÃO** (se existir), rejeitar com HTTP 409 Conflict. |
| **RN-02** | **SE** o modo for `CADASTRO` **ENTÃO** validar idade contra `configuracoes_regras_negocio` baseado em `tipo_registro` e `parentesco`. **SENÃO** (se modo `EDICAO`), desativar validação. **EXCEÇÃO:** Se `data_nascimento` for alterada na edição, reaplicar validação rigorosamente. |
| **RN-03** | **SE** `tipo_vinculo` = `SUB_DEPENDENTE` **ENTÃO** `agregado_pai_id` deve ser obrigatório e referenciar um `AGREGADO` com `estado_civil` IN (`CASADO`, `UNIAO_ESTAVEL`). **SENÃO**, rejeitar com HTTP 400. |
| **RN-04** | **SE** `estado_civil` de `AGREGADO` mudar de `CASADO`/`UNIAO_ESTAVEL` para `SOLTEIRO`/`DIVORCIADO`/`VIUVO` **ENTÃO** desvincular logicamente (`data_fim_vinculo`) APENAS sub-dependentes com `parentesco` = `CONJUGE`. **SENÃO** (se `FILHO`), manter vínculo ativo. |
| **RN-05** | **SE** `plano.tipo` = `INDIVIDUAL` **ENTÃO** bloquear cadastro de `DEPENDENTE`. **SENÃO** (se `FAMILIAR`), permitir até 8 dependentes. |
| **RN-06** | **SE** idade de `FILHO` > `IDADE_COBERTURA_FILHO` e <= `IDADE_LIMITE_DEPENDENTE` **ENTÃO** permitir cadastro com tag `"SEM DIREITO À PROTEÇÃO"`. **SENÃO** (se > limite), bloquear. |

### 4.2 Critérios de Aceite (Gherkin)

### 4.4 Edge Cases e Exceções (Tratamento de Falhas)
| Cenário | Comportamento Esperado | Ação do Sistema |
| :--- | :--- | :--- |
| **Injeção de `tipo_registro`** | Frontend manipulado envia `AGREGADO` na rota de `DEPENDENTE`. | Backend valida `tipo_registro` vs Rota. Retorna `400 Bad Request`. |
| **Concorrência de CPF** | Dois cadastros simultâneos para o mesmo CPF raro. | Constraint `UNIQUE` em `pessoas_fisicas.cpf`. Segundo request falha com `409 Conflict`. |
| **Timeout ViaCEP** | API externa de CEP não responde em 2s. | Frontend habilita campos de endereço para digitação manual. Não bloqueia o fluxo. |
| **Agregado sem Profissão** | Payload de Agregado omitindo `profissao_id`. | Backend rejeita (Campo obrigatório para `AGREGADO` e `TITULAR`). Retorna `422 Unprocessable Entity`. |
| **CPF com Letras (Futuro)** | Usuário digita CPF com caracteres alfabéticos. | Frontend sanitiza (ou permite se regex atualizada). Backend armazena como `VARCHAR`. |
| **Sub-dependente Órfão** | Agregado pai é deletado/excluído logicamente antes da submissão. | Backend valida integridade referencial em `vinculos`. Rejeita com `400`. |
| **Limite Global de Agregados** | Contrato já possui 4 agregados. Tentativa de adicionar o 5º. | Backend conta vínculos ativos. Rejeita com `400` "Limite de 4 agregados por contrato excedido". |
| **Retry de Webhook Financeiro** | Gateway reenvia confirmação de pagamento já processado. | Tabela `webhooks_recebidos` detecta duplicidade via constraint `(transaction_id, source)`. Retorna 200 OK sem reprocessar. |
| **Abatimento Duplicado** | Usuário clica duas vezes em "Abater Parcela". | Constraint `idx_abatimento_unico` impede segunda inserção. Retorna erro de integridade mapeado para 409. |

### 4.5 Tabela de Validações
| Campo | Regra de Validação | Mensagem de Erro | HTTP Code |
| :--- | :--- | :--- | :--- |
| `cpf` | Algorítmica + Unique no DB | "CPF inválido ou já cadastrado." | 400 / 409 |
| `data_nascimento` | `<= NOW()`, Respeita `IDADE_LIMITE_*` | "Idade máxima para {tipo} é {limite} anos." | 400 |
| `email` | RFC 5322, Lowercase | "Formato de e-mail inválido." | 400 |
| `telefone` | 10 ou 11 dígitos (apenas números) | "Telefone deve conter DDD + Número." | 400 |
| `cep` | 8 dígitos, formato válido | "CEP inválido." | 400 |
| `tipo_registro` | Deve corresponder à rota da API | "Tipo de registro incompatível com a operação." | 400 |
| `parentesco` | Restrito por `tipo_vinculo` | "Parentesco inválido para este tipo de vínculo." | 400 |

## 5. Especificação Não-Funcional

### 5.1 Performance
-   **Latência:** API de busca (`/buscar-por-cpf`) deve responder em `< 100ms` (p95).
-   **Throughput:** Suportar 500 requisições simultâneas de submissão de contratos sem degradação.
-   **Carga Frontend:** O componente `PessoaFisicaForm` deve pesar `< 40KB` (gzipped) e renderizar em `< 200ms` (LCP).

### 5.2 Escalabilidade
-   **Banco de Dados:** Tabelas `pessoas_fisicas` e `vinculos` devem ser indexadas por `cpf` e `titular_raiz_id`.
-   **Cache:** Configurações de regras de negócio (`configuracoes_regras_negocio`) devem ser cacheadas no Redis (TTL: 5 min).

### 5.3 Resiliência
-   **Retry Policy:** API ViaCEP possui retry de 2 tentativas com backoff exponencial (500ms, 1000ms).
-   **Circuit Breaker:** Se a API de Configurações cair, o Frontend usa o último snapshot válido.
-   **Fallback:** Em caso de falha na geração de PDF (pós-aprovação), o sistema enfileira a tarefa (Dead Letter Queue).

### 5.4 Segurança
-   **Sanitização:** Backend aplica `trim()`, remove espaços múltiplos e caracteres de controle.
-   **AuthN/AuthZ:** Rotas de submissão exigem Token JWT válido. Rota de listagem de seguradoras exige Role `SUPERADMIN`.
-   **Criptografia:** Dados sensíveis em trânsito (TLS 1.3). Hash de documentos em S3 (SHA-256).

### 5.5 Observabilidade
-   **Métricas (Prometheus):** `contratos_submissao_total`, `cpf_busca_duracao_ms`, `validacao_idade_bloqueio_total`.
-   **Logs (Datadog/ELK):** Logs estruturados (JSON) contendo `trace_id`, `user_id`, `action="SUBMIT_CONTRATO"`. **NUNCA** logar CPF ou Dados de Saúde em claro.
-   **Alertas:** Pico de erros 409 (Conflito de CPF) ou 500 na API de submissão.

### 5.6 Acessibilidade e i18n
-   **A11y:** WAI-ARIA 1.2. Campos obrigatórios com `aria-required="true"`. Erros anunciados por `aria-live="assertive"`.
-   **i18n:** Preparado para PT-BR e ES. Moeda formatada via `Intl.NumberFormat`.

## 6. Interface e Contratos

### 6.1 API (OpenAPI 3.0 - Trechos Críticos)
```yaml
paths:
  /api/pessoas-fisicas/buscar-por-cpf:
    get:
      summary: Busca pessoa física por CPF (Identificação Unificada)
      parameters:
        - name: cpf
          in: query
          required: true
          schema: { type: string }
      responses:
        '200': { description: CPF encontrado }
        '404': { description: CPF não encontrado (Fluxo de novo cadastro) }

  /api/contratos:
    post:
      summary: Submissão completa de contrato e árvore de vínculos
      requestBody:
        required: true
        content:
          application/json:
            schema:
              type: object
              properties:
                plano_id: { type: string, format: uuid }
                titular: { $ref: '#/components/schemas/PessoaFisicaPayload' }
                dependentes: { type: array, items: { $ref: '#/components/schemas/PessoaFisicaPayload' } }
                agregados: 
                  type: array
                  items: 
                    allOf:
                      - $ref: '#/components/schemas/PessoaFisicaPayload'
                      - type: object
                        properties:
                          sub_dependentes: { type: array, items: { $ref: '#/components/schemas/PessoaFisicaPayload' } }
      responses:
        '201': { description: Contrato criado (Status: AGUARDANDO_APROVACAO) }
        '400': { description: Erro de validação }
        '409': { description: Titular já possui contrato ativo }
```

### 6.2 Banco de Dados (Schema & Índices)
> 🚨 **ATENÇÃO:** As tabelas financeiras abaixo foram harmonizadas com o SPEC-03. Qualquer divergência deve ser tratada como bug crítico.

#### Tabelas de Cadastro (Core)
```sql
-- Tabela: pessoas_fisicas
CREATE TABLE pessoas_fisicas (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    nome_completo VARCHAR(255) NOT NULL,
    data_nascimento DATE NOT NULL,
    cpf VARCHAR(14) NOT NULL UNIQUE,
    genero CHAR(1) CHECK (genero IN ('M', 'F')),
    estado_civil VARCHAR(20) NOT NULL DEFAULT 'SOLTEIRO',
    tipo_registro VARCHAR(20) NOT NULL CHECK (tipo_registro IN ('TITULAR', 'DEPENDENTE', 'AGREGADO', 'SUB_DEPENDENTE')),
    titular_raiz_id UUID REFERENCES pessoas_fisicas(id),
    created_at TIMESTAMP DEFAULT NOW(),
    updated_at TIMESTAMP DEFAULT NOW()
);

-- Tabela: vinculos
CREATE TABLE vinculos (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    titular_raiz_id UUID NOT NULL REFERENCES pessoas_fisicas(id),
    pessoa_vinculada_id UUID NOT NULL REFERENCES pessoas_fisicas(id),
    tipo_vinculo VARCHAR(20) NOT NULL,
    parentesco VARCHAR(20) NOT NULL,
    agregado_pai_id UUID REFERENCES pessoas_fisicas(id),
    data_inicio_vinculo DATE NOT NULL DEFAULT CURRENT_DATE,
    data_fim_vinculo DATE,
    CONSTRAINT chk_dependente_parentesco CHECK (tipo_vinculo != 'DEPENDENTE' OR parentesco IN ('CONJUGE', 'FILHO')),
    CONSTRAINT chk_sub_dependente_parentesco CHECK (tipo_vinculo != 'SUB_DEPENDENTE' OR parentesco IN ('CONJUGE', 'FILHO')),
    CONSTRAINT chk_sub_dependente_pai CHECK (tipo_vinculo != 'SUB_DEPENDENTE' OR agregado_pai_id IS NOT NULL)
);
```

#### 🚨 Tabelas Financeiras Harmonizadas (Módulo 3)
```sql
-- Tabela: carteiras_digitais
CREATE TABLE carteiras_digitais (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    titular_id UUID NOT NULL REFERENCES pessoas_fisicas(id) UNIQUE,
    saldo_disponivel DECIMAL(12,2) NOT NULL DEFAULT 0,
    saldo_bloqueado DECIMAL(12,2) NOT NULL DEFAULT 0,
    saldo_devedor DECIMAL(12,2) NOT NULL DEFAULT 0 -- 🚨 NOVO CAMPO CRÍTICO
);

-- Tabela: transacoes_pagamento (Unificada)
CREATE TABLE transacoes_pagamento (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    carteira_id UUID NOT NULL REFERENCES carteiras_digitais(id),
    conta_a_pagar_id UUID REFERENCES contas_a_pagar(id),
    contrato_id UUID REFERENCES contratos(id),
    tipo_transacao VARCHAR(20) NOT NULL CHECK (tipo_transacao IN ('ABATIMENTO', 'SAQUE')),
    valor_abatido DECIMAL(10,2) NOT NULL,
    saldo_restante_parcela DECIMAL(10,2),
    valor_irrf_retido DECIMAL(10,2), -- 🚨 NOVO CAMPO
    valor_inss_retido DECIMAL(10,2), -- 🚨 NOVO CAMPO
    status VARCHAR(20) NOT NULL CHECK (status IN ('CONCLUIDO', 'ESTORNADO', 'PENDENTE_APROVACAO')),
    data_transacao TIMESTAMP DEFAULT NOW()
);
-- Constraint de Idempotência de Abatimento
CREATE UNIQUE INDEX idx_abatimento_unico ON transacoes_pagamento (conta_a_pagar_id, carteira_id) 
WHERE tipo_transacao = 'ABATIMENTO' AND status = 'CONCLUIDO';

-- Tabela: contas_a_pagar
CREATE TABLE contas_a_pagar (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    descricao VARCHAR(255) NOT NULL,
    valor DECIMAL(12,2) NOT NULL,
    valor_restante DECIMAL(12,2) NOT NULL,
    data_vencimento DATE NOT NULL,
    status VARCHAR(20) NOT NULL,
    boleto_url TEXT, -- 🚨 NOVO CAMPO
    gateway_transaction_id VARCHAR(100) -- 🚨 NOVO CAMPO
);

-- Tabela: webhooks_recebidos (Resiliência)
CREATE TABLE webhooks_recebidos (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    transaction_id VARCHAR(100) NOT NULL,
    source VARCHAR(50) NOT NULL,
    payload JSONB NOT NULL,
    processed_at TIMESTAMP DEFAULT NOW(),
    CONSTRAINT unique_webhook UNIQUE (transaction_id, source) -- 🔒 ANTI-DUPLICIDADE
);
```

## 7. Estratégia de Entrega e Mitigação de Riscos

### 7.1 Plano de Rollback/Migração
-   **Migração de Dados:** Greenfield para este módulo.
-   **Rollback de Deploy:** Reversão imediata do container. Scripts de rollback (`VX__down.sql`) testados em staging.
-   **Feature Flag:** `spf_new_contratacao_flow`. Se desligada, tráfego roteado para versão legada ou tela de manutenção.

### 7.2 Matriz de Riscos
| Risco | Probabilidade | Impacto | Plano de Mitigação |
| :--- | :--- | :--- | :--- |
| **Violação de Compliance SUSEP** | Baixa | Crítico | Code Review obrigatório com foco em ADR-002. Teste E2E verificando ausência de `seguradora_id` no DOM. |
| **Inconsistência de Vínculos** | Média | Alto | Constraints de FK e `CHECK` no DB. Job diário de saneamento. |
| **Fraude de Auto-Patrocínio** | Média | Alto | Validação Cross-Tree na camada de serviço antes de persistir `patrocinios`. |
| **Divergência de Schema Financeiro** | Baixa | Crítico | 🚨 **Mitigado nesta versão.** Schema financeiro copiado integralmente do SPEC-03 para a Seção 6.2. |
| **Latência Alta na Submissão** | Média | Médio | Transação de DB otimizada (Bulk Insert). Timeout de 5s no Frontend. |

### 7.3 Dependências Externas
-   **ViaCEP:** Instabilidade mitigada por fallback de input manual.
-   **S3:** Indisponibilidade bloqueia validação de usuários. Mitigado por retry e fila.
