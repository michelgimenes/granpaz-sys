## 2. Contexto e Escopo

### Problem Statement
A *hardcoding* de regras de negócio (idades, prazos de carência, limites financeiros) em código-fonte gera retrabalho de deploy para mudanças simples e risco de desalinhamento entre ambientes, violando o princípio de agilidade operacional. Adicionalmente, a ausência de um *Audit Trail* (trilha de auditoria) imutável e centralizado impede a rastreabilidade forense de alterações cadastrais e financeiras, expondo a administradora a sanções da SUSEP, LGPD e Receita Federal em caso de disputas judiciais ou auditorias fiscais.

Sob a ótica da integridade de dados, a falta de ferramentas de diagnóstico proativo permite que inconsistências críticas (órfãos de vínculo, ciclos de patrocínio, violações de Air-Gap clínico) persistam no banco de dados, degradando a confiabilidade do ecossistema multinível e de sinistros. Este módulo estabelece o motor de configurações dinâmicas tipadas, o repositório de logs imutáveis (WORM) e o painel de saneamento de integridade referencial.

### Objetivos
| Categoria | Métrica / KPI | Alvo |
| :--- | :--- | :--- |
| **Performance** | Latência p99 de leitura de configurações (`GET /configuracoes`) | `< 10ms` (Cache Redis Hit) |
| **Compliance** | Taxa de alterações cadastrais/financeiras sem registro em `audit_logs` | `0%` (Cobertura 100%) |
| **Integridade** | Tempo de execução do Job de Diagnóstico de Integridade (Full Scan) | `< 5 minutos` |
| **Segurança** | Detecção de vocabulário CLT (Air-Gap Funcional) em logs/metadata | `< 1 hora` (Alerta Automático) |
| **Operacional** | Tempo de propagação de nova configuração para o Frontend (Invalidação) | `< 10 segundos` |

### Out of Scope (Não será implementado nesta versão)
1.  **Motor de Regras Complexo (Drools/Rego):** O sistema suporta apenas pares Chave-Valor tipados, não árvores de decisão lógicas.
2.  **BI e Analytics:** Logs de auditoria são para forense e compliance, não para dashboards de negócio (PowerBI/Tableau).
3.  **Logs de Aplicação (APM):** Traces de erro e performance (Datadog/New Relic) são infraestrutura, não domínio deste módulo.
4.  **Gestão de Papéis e Permissões (RBAC Core):** Apenas consome os tokens JWT; o cadastro de usuários reside no Módulo de Auth.
5.  **Versionamento de Markdown de Seguradoras:** O histórico de cláusulas é tratado no Módulo 2 (Backoffice).

### Premissas e Restrições
├── **Stack:** Node.js/TypeScript, PostgreSQL 15+ (JSONB, Triggers), Redis (Cache), AWS S3 (WORM para logs críticos).
├── **Compliance:** LGPD (Art. 37 - Encarregado e Registros), SUSEP (Circular 666/2022 - Rastreabilidade), Código Civil.
├── **Imutabilidade:** Registros de auditoria (`audit_logs`) são *Append-Only*. `UPDATE` e `DELETE` são bloqueados via Trigger de Banco e IAM restrito no S3.
└── **Verdade Absoluta:** As 16 chaves de configuração listadas neste documento são mandatórias. A ausência de qualquer uma em ambiente de produção é classificada como **Bug Severidade 1**.

### Glossário
| Termo | Definição |
| :--- | :--- |
| **Audit Trail** | Registro cronológico e imutável de quem alterou o quê, quando e de qual origem. |
| **WORM** | *Write Once, Read Many*. Storage configurado para impedir exclusão ou sobrescrita de dados. |
| **Air-Gap Funcional** | Isolamento conceitual que proíbe a existência de termos ou lógicas de subordinação trabalhista (CLT) no sistema. |
| **Diagnóstico de Integridade** | Queries analíticas que identificam violações de regras de negócio que contornaram as constraints de banco. |
| **Stale Cache** | Dados desatualizados no Redis ou Frontend após uma alteração no banco de origem. |

### 3.3 Architecture Decision Records (ADRs)
| ID | Decisão | Alternativas Consideradas | Justificativa | Consequências |
| :--- | :--- | :--- | :--- | :--- |
| **ADR-01** | **Cache Multi-Camada (Redis + Frontend SW)** | Apenas DB ou apenas Redis. | Configurações de UX (idades, máscaras) são lidas a cada renderização de formulário. Cache no Frontend (TTL 5 min) reduz carga do Redis e latência de rede. | Exige mecanismo de invalidação via WebSocket/Event Bus para evitar *Stale Cache*. |
| **ADR-02** | **Logs de Auditoria via Camada de Serviço** | Triggers nativas do PostgreSQL. | Triggers dificultam a captura de metadados de requisição (IP, User-Agent, Trace ID) e o roteamento para S3 WORM. Service Layer centraliza a lógica de formatação JSONB. | Exige disciplina rigorosa: toda alteração de domínio deve passar pelo Service de Auditoria. |
| **ADR-03** | **Parser de Tipagem Forte na Service** | Salvar tudo como `VARCHAR` e parsear no consumo. | Previne que um Admin insira "vinte" em um campo de idade (INT), quebrando jobs noturnos de suspensão ou maioridade. A validação ocorre na escrita. | O schema do DB armazena `VARCHAR`, mas a API rejeita payloads que não match o tipo declarado no metadado da chave. |
| **ADR-04** | **Air-Gap Funcional via NLP Simples** | Ignorar vocabulário de RH. | 🚨 **COMPLIANCE TRABALHISTA:** A detecção de termos como "meta", "ponto" ou "salário" em logs de suporte ou descrições de chamados previne passivos de vínculo empregatício. | Job diário varre `audit_logs` e `logs_suporte` buscando regex de termos proibidos. |

## 4. Especificação Funcional Detalhada

### 4.1 Regras de Negócio (Determinísticas)
| ID | Regra de Negócio (SE / ENTÃO / SENÃO) |
| :--- | :--- |
| **RN-01** | 🔒 **SE** tentativa de atualizar `configuracoes_regras_negocio` **ENTÃO** o sistema deve validar o tipo do `valor` contra o `tipo_parse` definido no metadado da chave (ex: INT, BOOLEAN, DECIMAL). **SENÃO** (se tipo inválido), rejeitar com `400 Bad Request`. |
| **RN-02** | 🛡️ **SE** qualquer entidade de domínio (`pessoas_fisicas`, `contratos`, `patrocinios`) sofrer `UPDATE` **ENTÃO** o sistema deve inserir registro em `audit_logs` contendo `valores_anteriores` e `valores_novos` (JSONB). **EXCEÇÃO:** Campos de alta rotatividade e baixo risco (ex: `updated_at`, `ultimo_login`) são ignorados via whitelist. |
| **RN-03** | ⚠️ **SE** Job de Diagnóstico identificar `pessoas_fisicas` com `tipo_registro` = `DEPENDENTE` órfão (sem `vinculos` ativo e sem `titular_raiz_id` válido) **ENTÃO** logar em `alertas_integridade` e notificar o Suporte Técnico. **SENÃO** (se for Titular), ignorar. |
| **RN-04** | 🚨 **SE** Job de Air-Gap Funcional detectar termos CLT (`meta`, `horario`, `ponto`, `chefe`, `salario`, `jornada`) em `audit_logs.observacoes` ou metadados de UI **ENTÃO** gerar alerta crítico de Compliance Trabalhista para o DPO e Jurídico. **EXCEÇÃO:** Nenhuma. |
| **RN-05** | 💰 **SE** a chave `HASH_ASSINATURA_PDF_SALT` for alterada **ENTÃO** o sistema deve invalidar globalmente todos os caches de PDF e forçar a regeração de certificados em background. **SENÃO** (se qualquer outra chave), invalidar apenas o cache específico. |

### 4.4 Edge Cases e Exceções (30% da Seção - Foco Crítico)
| Cenário | Condição / Falha | Comportamento Esperado do Sistema |
| :--- | :--- | :--- |
| **EC-01: Concorrência na Atualização** | Dois Admins alteram a mesma chave simultaneamente. | Lock otimista via `updated_at` ou Lock pessimista (`SELECT FOR UPDATE`). O segundo request falha com `409 Conflict` ("Configuração alterada por outro usuário. Recarregue."). |
| **EC-02: Redis Indisponível** | Cache de configurações cai durante pico de acessos. | Fallback automático para leitura direta no PostgreSQL (Read Replica). Alerta de degradação de performance disparado. Frontend usa *stale-while-revalidate* do Service Worker. |
| **EC-03: Injeção de JSON Malicioso** | Admin tenta injetar payload JSONB gigante em `valores_novos` para estourar limite de disco. | API limita o tamanho do JSONB de auditoria a `10KB`. Se exceder, trunca campos não-essenciais e loga alerta de tentativa de *Log Injection*. |
| **EC-04: Falha no Backup WORM** | S3 Bucket de auditoria retorna `503 Service Unavailable`. | A transação de DB **não** é abortada (para não bloquear o negócio), mas o evento de log é enfileirado em DLQ (Dead Letter Queue) para retry infinito até o S3 recuperar. Alerta Crítico no PagerDuty. |
| **EC-05: Diagnóstico de Ciclos Infinitos** | Job de integridade encontra ciclo em `patrocinios` (A->B->C->A). | O Job não tenta corrigir automaticamente (risco de quebrar árvore). Ele isola o ciclo, marca os nós como `INTEGRIDADE_VIOLADA` e notifica o SuperAdmin para intervenção manual via Módulo 5. |
| **EC-06: Alteração de Chave Crítica (`PREVALENCIA_APIOLICE`)** | Admin desativa a flag `PREVALENCIA_APIOLICE_SOBRE_CONFIG`. | O sistema exige **Dupla Aprovação** (Maker/Checker). Um Admin solicita, outro (ou o Jurídico) aprova. Log de auditoria registra explicitamente o risco assumido. |

### 4.5 Validações de Entrada
| Campo | Regra | Mensagem de Erro | HTTP Code |
| :--- | :--- | :--- | :--- |
| `chave` | Deve existir no Enum de Chaves Obrigatórias | "Chave de configuração desconhecida." | 400 |
| `valor` | Deve match o `tipo_parse` da chave | "Tipo inválido para esta chave." | 400 |
| `motivo_alteracao` | String, min 20 chars, max 500 chars | "Motivo da alteração é obrigatório para auditoria." | 400 |

## 5. Especificação Não-Funcional

### 5.1 Performance
├── **Latência:** Leitura de configurações (`GET /configuracoes`) `< 10ms` (p95) via Redis.
├── **Throughput:** Suportar 1.000 requisições/segundo de validação de regras no Frontend sem atingir o DB.
├── **Diagnóstico:** Job de Integridade Noturno deve processar 1M de registros em `< 10 minutos` usando *Read Replicas* e *Cursors*.
└── **Cache Hit Rate:** Meta de `> 95%` para leituras de configuração.

### 5.2 Escalabilidade
├── **Redis:** Cluster mode (3 shards) para garantir alta disponibilidade do cache de regras.
├── **DB:** Particionamento da tabela `audit_logs` por mês (`RANGE PARTITION BY created_at`) para manter queries rápidas e facilitar o arquivamento (S3) de logs antigos.
└── **Frontend:** Service Worker intercepta requests de `/configuracoes` e serve do Cache API do navegador.

### 5.3 Resiliência
├── **SLO:** `99.99%` de disponibilidade para leitura de configurações (crítico para UX de cadastro).
├── **Circuit Breaker:** Se o S3 WORM falhar, o sistema continua operando, mas bloqueia alterações de configurações críticas (ex: `HASH_ASSINATURA_PDF_SALT`) até a restauração do backup.
├── **Retry Policy:** Logs de auditoria falhos são retentados com backoff exponencial (1s, 5s, 30s, 5m) antes de irem para DLQ.

### 5.4 Segurança
├── **AuthN/AuthZ:** Apenas tokens com role `SUPERADMIN` podem executar `PUT /configuracoes`. Roles `SUPORTE` e `FINANCEIRO` têm acesso *Read-Only* a logs de auditoria de seus respectivos domínios.
├── **Imutabilidade (WORM):** Bucket S3 configurado com *Object Lock* (Governance Mode) por 5 anos. Nem o root user da AWS pode excluir logs antes do prazo legal.
├── **Sanitização:** Valores de `audit_logs` passam por escape de HTML para prevenir *Stored XSS* no painel de Backoffice.
└── **Air-Gap Funcional:** Regex de bloqueio de termos CLT aplicada em todos os campos de texto livre do sistema (motivos, observações, descrições).

### 5.5 Observabilidade
├── **Métricas (Prometheus):**
│   ├── `config_cache_hit_total` vs `config_cache_miss_total`
│   ├── `audit_log_insertion_duration_seconds`
│   └── `integridade_job_violacoes_total{tipo="orfao|ciclo|air_gap"}`
├── **Logs Estruturados:** JSON contendo `trace_id`, `ator_id`, `entidade`, `acao="UPDATE"`, `ip_address`.
└── **Alertas (PagerDuty):**
    ├── Falha na inserção de Log de Auditoria (Crítico - Risco Legal).
    ├── Detecção de Termo CLT em logs (Crítico - Risco Trabalhista).
    └── Cache Hit Rate < 80% por mais de 5 minutos.

### 5.6 Acessibilidade e i18n
├── **A11y:** Painel de Auditoria deve suportar navegação por teclado e leitura de tabelas complexas via Screen Reader (ARIA labels em colunas JSONB).
└── **i18n:** Chaves de configuração são traduzidas no Frontend (`config.IDADE_LIMITE_TITULAR` -> "Idade Máxima do Titular"). O DB armazena apenas a chave técnica.

## 7. Estratégia de Entrega e Mitigação de Riscos

### 7.1 Plano de Rollback/Migração
├── **Migração:** Script de *Seed* populando as 16 chaves obrigatórias com valores padrão extraídos da Verdade Absoluta.
├── **Backfill de Auditoria:** Job único para gerar logs de auditoria sintéticos (`acao = 'MIGRATION'`) para registros críticos pré-existentes.
└── **Rollback:** Reversão do container. A tabela `audit_logs` é *Append-Only*, então rollbacks de código não afetam a integridade histórica dos logs.

### 7.2 Feature Flags
├── `AUDIT_LOG_WORM_ENABLED`: Se `true`, replica logs para S3 WORM. Se `false`, apenas DB local (para testes de performance).
├── `AIR_GAP_NLP_SCANNER_ENABLED`: Ativa o job de varredura de vocabulário CLT em logs de suporte.

### 7.3 Matriz de Riscos
| Risco | Probabilidade | Impacto | Plano de Mitigação |
| :--- | :--- | :--- | :--- |
| **Perda de Logs de Auditoria** | Baixa | Crítico (Legal) | S3 WORM com Object Lock. DLQ para retries de rede. Alerta imediato se inserção falhar. |
| **Stale Cache no Frontend** | Média | Alto (UX/Regra) | Websocket + Event Bus para invalidação forçada do Service Worker. Fallback de TTL curto (5 min). |
| **Injeção de Termos CLT** | Média | Crítico (Trabalhista) | Job de NLP/Regex varrendo `audit_logs` e `logs_suporte` diariamente. Alerta para o DPO. |
| **Alteração Maliciosa de Chave Crítica** | Baixa | Alto (Financeiro) | Chaves como `HASH_ASSINATURA_PDF_SALT` exigem Maker/Checker (Dupla Aprovação). |

### 7.4 Dependências Externas
├── **Módulo 1 a 5:** Todos dependem da API `GET /configuracoes` para validação de regras de negócio.
├── **AWS S3:** Dependência crítica para storage WORM de logs de auditoria.
├── **Redis:** Dependência crítica para performance de leitura de configurações.

## 8. Checklist de Validação do SPEC

- [ ] **Imutabilidade:** Triggers de banco impedem `UPDATE` e `DELETE` na tabela `audit_logs`?
- [ ] **Air-Gap Funcional:** O sistema possui mecanismo de detecção de termos CLT ("meta", "horário", "chefe") em campos de texto livre?
- [ ] **Cache Invalidation:** A atualização de uma configuração dispara evento para invalidar o Redis e o Service Worker do Frontend?
- [ ] **Tipagem Forte:** A API valida o tipo do valor (INT, BOOLEAN) contra o metadado da chave antes de persistir?
- [ ] **WORM Storage:** O bucket S3 de logs está configurado com *Object Lock* (Governance Mode)?
- [ ] **Diagnóstico:** Os Jobs de Integridade (Órfãos, Ciclos, Air-Gap Clínico) estão agendados e possuem alertas configurados?
- [ ] **Compliance SUSEP/LGPD:** As 16 chaves obrigatórias estão presentes no script de Seed inicial?
- [ ] **QA Testability:** Os cenários Gherkin cobrem a invalidação de cache e a tentativa de injeção de tipos inválidos?
