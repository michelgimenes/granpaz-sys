## 2. Contexto e Escopo

### Problem Statement

O módulo financeiro do Sistema de Gestão Saúde & Proteção (SPF) enfrenta três problemas críticos que impedem a escalabilidade do modelo de negócio:

**1. Fragilidade Financeira e Risco de Fraude:** O sistema atual não possui idempotência robusta em transações de bonificação multinível, resultando em duplicidade de pagamentos em cenários de retry de rede ou falhas parciais. Além disso, a ausência de controles de saldo devedor permite que revendedores realizem saques ou abatimentos mesmo após cancelamentos de contratos que deveriam estornar bonificações já liberadas.

**2. Inconsistência Tributária e Compliance:** A mistura de receita tributável (clube de benefícios) com repasse de risco (prêmio de seguro) em uma única transação financeira gera exposição fiscal (ISS indevido sobre prêmios) e viola diretrizes SUSEP. O sistema não possui separação contábil clara, nem retenção automática de IRRF/INSS em saques de Pessoa Física, criando passivos trabalhistas e tributários.

**3. Opacidade Operacional:** A falta de reconciliação automática entre boletos pagos, bonificações liberadas e abatimentos via carteira digital impede auditoria financeira confiável. O time financeiro não possui visibilidade em tempo real de saldos devedores, transações pendentes de aprovação manual ou conflitos de rede (CPF ativo em múltiplas árvores de patrocínio).

### Objetivos

#### KPIs de Negócio

| Métrica | Meta | Justificativa |
|---------|------|---------------|
| **Redução de Estornos Manuais** | < 0.5% do total de transações/mês | Minimizar intervenção humana e erro operacional |
| **Tempo Médio de Liberação de Bonificação** | < 2s após aprovação de contrato | Garantir experiência fluida para revendedores |
| **Acurácia de Reconciliação Financeira** | 100% (zero divergências) | Compliance fiscal e auditoria SUSEP |
| **Taxa de Saques Bloqueados por Saldo Devedor** | 100% dos casos aplicáveis | Prevenção de fraude e perda financeira |

| **Disponibilidade (SLA)** | 99.95% | Crítico para operações financeiras |
| **Taxa de Erros (5xx)** | < 0.1% | Confiabilidade do sistema |

### Glossário

| Termo | Definição | Exemplo |
|-------|-----------|---------|
| **Bonificação** | Remuneração paga a revendedores por indicação/fechamento de novos clientes. **NUNCA** usar "comissão" ou "comissionamento" para evitar caracterização de corretagem (SUSEP) | R$ 50,00 por novo contrato |
| **Carteira Digital** | Conta virtual do revendedor com saldos: disponível, bloqueado, devedor | Saldo Disponível: R$ 500,00 |
| **Estorno** | Reversão de bonificação já liberada, debitando do saldo disponível (pode gerar saldo devedor) | Contrato cancelado → estorna R$ 50,00 |
| **Abatimento** | Uso de saldo disponível para quitar parcelas de contratos próprios ou de terceiros | Revendedor paga parcela de R$ 100,00 com saldo |
| **Saldo Devedor** | Valor negativo na carteira após estorno que excedeu saldo disponível | Saldo: -R$ 100,00 (bloqueia novos saques) |
| **Idempotência** | Garantia de que operação executada múltiplas vezes produz mesmo resultado | Constraint UNIQUE impede bonificação duplicada |
| **Maker/Checker** | Fluxo de aprovação manual para operações acima de limite configurado | Saque > R$ 1.000,00 exige aprovação do Financeiro |
| **RPA (Recibo de Pagamento Autônomo)** | Documento fiscal para pagamentos a Pessoa Física sem vínculo empregatício | Saque PF com retenção IRRF/INSS |
| **Air-Gap Financeiro** | Separação contábil estrita entre receita tributável (clube) e repasse de risco (prêmio) | NF emitida apenas sobre serviço, não sobre prêmio |

### Architecture Decision Records (ADRs)

| # | Decisão | Alternativas Consideradas | Justificativa | Consequências |
|---|---------|---------------------------|---------------|---------------|
| ADR-001 | **Usar PostgreSQL com constraints compostas para idempotência** | 1. Application-level locking (Redis)<br>2. Tabela de auditoria separada<br>3. Message queue com deduplication | PostgreSQL já garante ACID; constraints são mais rápidas que locks distribuídos; sem dependência externa | ✅ Zero duplicidade<br>⚠️ Exige migrations cuidadosas<br>✅ Auditoria nativa |
| ADR-002 | **Separar Financial Service e Bonus Service em microsserviços** | 1. Monolito único<br>2. Módulos dentro do mesmo serviço | Separação de responsabilidades; bonus tem lógica complexa de árvore; financial precisa de alta disponibilidade | ✅ Escalabilidade independente<br>⚠️ Complexidade de rede<br>✅ Times podem trabalhar em paralelo |
| ADR-003 | **Usar BullMQ para jobs assíncronos (geração de boletos, notificações)** | 1. RabbitMQ puro<br>2. AWS SQS + Lambda<br>3. Cron jobs nativos | BullMQ oferece retry policies, dead letter queues, rate limiting out-of-the-box; integração nativa com Node.js | ✅ Retry automático com backoff<br>⚠️ Dependência de Redis<br>✅ Dashboard de monitoramento |
| ADR-004 | **Implementar saldo devedor como campo explícito (não calculado)** | 1. Calcular em tempo real (SUM transações)<br>2. Bloquear operações que gerariam negativo | Performance (evita queries pesadas); clareza em auditoria; permite regras de priorização de abatimento futuro | ✅ Queries rápidas<br>⚠️ Exige trigger/job de atualização<br>✅ Histórico imutável |
| ADR-005 | **Não usar eventos assíncronos para liberação de bonificação (síncrono na aprovação)** | 1. Publicar evento e processar via worker<br>2. Webhook para serviço de bonus | Garante atomicidade: bonificação só é liberada se contrato for realmente aprovado; evita race conditions | ✅ Consistência imediata<br>⚠️ Acoplamento temporal<br>✅ Simplicidade de debugging |
| ADR-006 | **Cache de saldo de carteira no Redis com invalidation via eventos** | 1. Sempre consultar DB<br>2. Cache no frontend (localStorage) | Latência sub-millisecond para dashboard; evita sobrecarga do DB em picos | ✅ UX instantânea<br>⚠️ Complexidade de invalidation<br>⚠️ Risco de stale data se evento falhar |

## 4. Especificação Funcional Detalhada

### 4.1 Regras de Negócio (Formato SE-ENTÃO-SENÃO)

#### RN-001: Base de Cálculo da Bonificação

**SE** contrato for aprovado  
**ENTÃO** calcular bonificação aplicando `percentual_indicacao` ou `percentual_fechamento` (conforme nível na árvore) sobre:  
- Taxa de Adesão (se pagamento único)  
- Valor da Parcela Mensal/Anual  
**SENÃO** (se tentar incluir prêmio de seguro ou capital segurado)  
**EXCEÇÃO** Bloquear operação com erro `BONUS_BASE_CALCULO_INVALIDO` e logar auditoria

**Justificativa:** Compliance SUSEP - bonificação remunera atividade comercial, não intermediação de seguro

---

#### RN-002: Idempotência de Bonificação

**SE** sistema tentar criar `transacoes_bonificacao` com mesma combinação `(origem_contrato_id, carteira_id, nivel_origem)`  
**ENTÃO** PostgreSQL constraint UNIQUE impedirá inserção (violates unique constraint)  
**SENÃO** (se aplicação tentar retry sem verificar constraint)  
**EXCEÇÃO** Retornar `409 Conflict` com mensagem "Bonificação já processada para este contrato e nível"

**Implementação:**
```sql
CREATE UNIQUE INDEX idx_bonificacao_unica
ON transacoes_bonificacao (origem_contrato_id, carteira_id, nivel_origem);
```

---

#### RN-003: Abatimento de Parcela com Saldo Insuficiente

**SE** revendedor tentar abater parcela  
**E** `saldo_disponivel` <= 0  
**ENTÃO** bloquear operação e retornar erro `SALDO_INSUFICIENTE`  
**SENÃO** (se `saldo_disponivel` > 0 mas < `valor_parcela`)  
**EXCEÇÃO** Permitir abatimento parcial, gerar boleto residual com `valor_restante`

**Cálculo:**
```typescript
const valorAbatido = Math.min(saldoDisponivel, valorParcela);
const saldoRestante = valorParcela - valorAbatido;
const statusConta = saldoRestante === 0 ? 'PAGO' : 'PARCIALMENTE_PAGO';
```

---

#### RN-004: Estorno Pós-Aprovação (Cancelamento de Contrato)

**SE** contrato com status `APROVADO` for cancelado  
**ENTÃO** para cada `transacoes_bonificacao` com status `LIBERADO`:  
1. Alterar status para `ESTORNADO`  
2. Debitar `valor` de `saldo_disponivel` da carteira  
3. **SE** `saldo_disponivel` < 0 após débito:  
   - Transferir valor absoluto para `saldo_devedor`  
   - Zerar `saldo_disponivel`  
**SENÃO** (se bonificação já foi usada em `transacoes_pagamento`)  
**EXCEÇÃO** Manter débito negativo (saldo devedor), **NÃO** reverter pagamento da parcela (boleto já foi emitido/pago)

---

#### RN-005: Saque de Bonificação com Retenção Tributária (PF)

**SE** revendedor for Pessoa Física  
**E** `configuracoes_regras_negocio.SAQUE_PF_ATIVO = true`  
**E** `valor_saque` > 0  
**ENTÃO**:  
1. Calcular IRRF conforme tabela progressiva 2026  
2. Calcular INSS (11% até teto R$ 908,85 em 2026)  
3. Persistir `valor_irrf_retido` e `valor_inss_retido`  
4. Creditar apenas `valor_liquido = valor_bruto - irrf - inss`  
5. Gerar DARF/GPS no módulo Financeiro  
**SENÃO** (se `SAQUE_PF_ATIVO = false`)  
**EXCEÇÃO** Bloquear saque com erro `SAQUE_PF_DESATIVADO`

---

#### RN-006: Maker/Checker para Saques Acima do Limite

**SE** `valor_saque` > `configuracoes_regras_negocio.LIMITE_SAQUE_DIARIO`  
**ENTÃO**:  
1. Criar `transacoes_pagamento` com status `PENDENTE_APROVACAO`  
2. Notificar Financeiro via dashboard e email  
3. Bloquear crédito até aprovação manual  
**SENÃO** (se `valor_saque` <= limite)  
**EXCEÇÃO** Processar imediatamente (status `CONCLUIDO`)

---

#### RN-007: Bloqueio de Saque com Saldo Devedor

**SE** revendedor possuir `saldo_devedor` > 0  
**ENTÃO**:  
1. Bloquear novos saques  
2. Priorizar abatimento automático de futuras bonificações para cobrir saldo devedor antes de liberar `saldo_disponivel`  
**SENÃO** (se revendedor tentar saque mesmo com saldo devedor)  
**EXCEÇÃO** Retornar erro `SALDO_DEVEDOR_EXISTENTE` com mensagem: "Regularize seu saldo devedor de R$ X antes de realizar novos saques"

---

#### RN-008: Recorrência Automática (Geração de Parcelas)

**SE** job diário executar às 02:00 AM  
**E** contrato.status = `APROVADO`  
**E** contrato.periodicidade = `MENSAL`  
**E** última parcela quitada/vencida há > 5 dias  
**ENTÃO**:  
1. Recalcular valor baseado em agregados ativos no momento  
2. Criar nova `contas_a_pagar` com `data_vencimento` = próximo ciclo  
3. Gerar boleto e atualizar `contratos_legado_flat` (apenas campos financeiros)  
**SENÃO** (se contrato estiver `SUSPENSO`)  
**EXCEÇÃO** Não gerar nova parcela até regularização

---

#### RN-009: Anti-Conflito de Rede (Cross-Tree)

**SE** tentativa de cadastro de novo revendedor  
**E** CPF já existe em outra árvore de patrocínio ativa (`data_fim_vinculo IS NULL`)  
**ENTÃO** bloquear com erro `CONFLITO_DE_REDE`  
**SENÃO** (se CPF existe mas com `data_fim_vinculo` preenchido)  
**EXCEÇÃO** Permitir cadastro (revendedor inativo pode ser reativado em nova árvore)

---

#### RN-010: Air-Gap de Auto-Patrocínio

**SE** tentativa de criar vínculo de patrocínio  
**E** `revendedor_id` = `patrocinador_id`  
**OU** `revendedor_id` é dependente direto (cônjuge/filho) de `patrocinador_id`  
**ENTÃO** bloquear com erro `AUTO_PATROCINIO_PROIBIDO`  
**SENÃO** (se for parentesco indireto: irmão, tio, primo)  
**EXCEÇÃO** Permitir (mas logar auditoria para revisão manual)

### 4.4 Edge Cases e Exceções (30% da Seção)

#### 4.4.1 Cenários de Concorrência e Race Conditions

##### EC-001: Múltiplos Abatimentos Simultâneos na Mesma Carteira

**Cenário:** Revendedor abre duas abas do navegador e clica "Abater Parcela" simultaneamente em duas parcelas diferentes, ambas com valor R$ 300,00. Saldo disponível é R$ 500,00.

**Comportamento Esperado:**
1. Primeira requisição adquire lock distribuído no Redis (`SET carteira:{id}:lock NX PX 5000`)
2. Primeira requisição inicia transação PostgreSQL (`BEGIN`)
3. Primeira requisição consulta saldo com `SELECT saldo_disponivel FROM carteiras_digitais WHERE id = ? FOR UPDATE` (lock de linha)
4. Segunda requisição tenta adquirir lock no Redis e falha (timeout 5s)
5. Segunda requisição retorna `409 Conflict` com mensagem: "Operação em andamento. Tente novamente em alguns segundos."
6. Primeira requisição processa abatimento, commit, libera lock

**Implementação:**
```typescript
async function abaterParcela(carteiraId: string, contaId: string, valor: number) {
  const lockKey = `carteira:${carteiraId}:lock`;
  const lockAcquired = await redis.set(lockKey, '1', 'EX', 5, 'NX');
  
  if (!lockAcquired) {
    throw new ConflictException('Operação em andamento. Tente novamente.');
  }
  
  try {
    return await prisma.$transaction(async (tx) => {
      const carteira = await tx.carteirasDigitais.findUnique({
        where: { id: carteiraId },
        lock: 'UPDATE' // SELECT ... FOR UPDATE
      });
      
      if (carteira.saldoDisponivel < valor) {
        throw new BadRequestException('Saldo insuficiente');
      }
      
      // Processar abatimento...
      
      return { success: true };
    });
  } finally {
    await redis.del(lockKey);
  }
}
```

##### EC-002: Aprovação de Contrato e Cancelamento Simultâneos

**Cenário:** SuperAdmin clica "Aprovar" no contrato C500. No mesmo milissegundo, job de cancelamento por inadimplência tenta cancelar o mesmo contrato.

**Comportamento Esperado:**
1. Aprovação adquire lock de linha em `contratos` (`SELECT ... FOR UPDATE`)
2. Aprovação inicia transação e altera status para `APROVADO`
3. Job de cancelamento tenta adquirir lock e bloqueia (wait)
4. Aprovação commit e libera lock
5. Job de cancelamento adquire lock, verifica status = `APROVADO`, e **aborta** (não cancela contrato recém-aprovado)
6. Job loga: "Contrato C500 ignorado (status alterado durante processamento)"

**Constraint de Banco:**
```sql
-- Trigger para impedir cancelamento de contrato aprovado há menos de 1 minuto
CREATE OR REPLACE FUNCTION impedir_cancelamento_imediato()
RETURNS TRIGGER AS $$
BEGIN
  IF NEW.status = 'CANCELADO' 
     AND OLD.status = 'APROVADO'
     AND NOW() - OLD.updated_at < INTERVAL '1 minute'
  THEN
    RAISE EXCEPTION 'Contrato recém-aprovado não pode ser cancelado imediatamente';
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trigger_impedir_cancelamento
BEFORE UPDATE ON contratos
FOR EACH ROW
EXECUTE FUNCTION impedir_cancelamento_imediato();
```

##### EC-003: Webhook de Pagamento Duplicado (Gateway Retry)

**Cenário:** Gateway de pagamento envia webhook de confirmação de pagamento do boleto B100. Rede falha, gateway faz retry 3 vezes com mesmo payload.

**Comportamento Esperado:**
1. Primeiro webhook recebido: sistema cria `contas_a_pagar` com status `PAGO`, gera bonificação, etc.
2. Segundo webhook recebido: sistema verifica idempotency key (campo `gateway_transaction_id` na tabela `webhooks_recebidos`)
3. Sistema detecta duplicação e retorna `200 OK` (sem processar novamente)
4. Sistema loga: "Webhook duplicado ignorado (transaction_id: X)"

**Implementação:**
```typescript
// Middleware de idempotência
async function idempotencyMiddleware(req, res, next) {
  const idempotencyKey = req.headers['x-idempotency-key'] || req.body.transaction_id;
  
  if (!idempotencyKey) {
    return next();
  }
  
  const existing = await prisma.webhooksRecebidos.findUnique({
    where: { transactionId: idempotencyKey }
  });
  
  if (existing) {
    return res.status(200).json({ message: 'Already processed', duplicate: true });
  }
  
  await prisma.webhooksRecebidos.create({
    data: {
      transactionId: idempotencyKey,
      payload: req.body,
      processedAt: new Date()
    }
  });
  
  next();
}
```

```typescript
async function garantirCarteira(titularId: string): Promise<CarteiraDigital> {
  let carteira = await prisma.carteirasDigitais.findUnique({
    where: { titularId }
  });
  
  if (!carteira) {
    carteira = await prisma.carteirasDigitais.create({
      data: {
        titularId,
        saldoDisponivel: 0,
        saldoBloqueado: 0,
        saldoDevedor: 0
      }

  }

```

#### 4.4.3 Cenários de Timeout e Falhas de Dependências Externas

##### EC-006: Timeout na Geração de Boleto (Gateway Indisponível)

**Cenário:** Sistema tenta gerar boleto via API do gateway, mas gateway retorna timeout após 30s.

**Comportamento Esperado:**
1. Sistema tenta 3 vezes com backoff exponencial (1s, 2s, 4s)
2. Se todas falharem:
   - Sistema cria `contas_a_pagar` com status `PENDENTE` mas sem `boleto_url`
   - Sistema publica evento `BoletoGeracaoFalhou` na fila
   - Job de retry tenta novamente em 1h, 6h, 24h
   - Sistema envia email para titular: "Estamos com instabilidade na geração do seu boleto. Tentaremos novamente em breve."
   - Sistema notifica DevOps via PagerDuty
3. Frontend exibe: "Boleto em processamento. Você receberá por email em até 24h."

**Implementação:**
```typescript
const retryPolicy = {
  maxAttempts: 3,
  backoff: {
    type: 'exponential',
    delay: 1000 // 1s, 2s, 4s
  },
  retryableErrors: ['TIMEOUT', 'NETWORK_ERROR', '503']
};

async function gerarBoleto(contaId: string): Promise<string> {
  try {
    return await retryWithBackoff(
      () => paymentGateway.createBoleto(contaId),
      retryPolicy
    );
  } catch (error) {
    await prisma.contasAPagar.update({
      where: { id: contaId },
      data: { status: 'PENDENTE', boletoUrl: null }
    });
    
    await messageBroker.publish('BoletoGeracaoFalhou', { contaId, error: error.message });
    await notificationService.sendEmail(titularId, 'boleto_pendente');
    
    throw new ServiceUnavailableException('Falha ao gerar boleto');
  }
}
```

##### EC-007: Redis Indisponível (Cache de Saldos)

**Cenário:** Redis cluster cai durante pico de acesso ao dashboard de revendedores.

**Comportamento Esperado:**
1. Sistema detecta falha de conexão com Redis (timeout 100ms)
2. Sistema ativa modo "fallback para DB"
3. Todas as consultas de saldo vão direto para PostgreSQL (mais lento, mas funcional)
4. Sistema loga warning: "Redis indisponível, usando fallback para DB"
5. Sistema envia alerta para DevOps (PagerDuty nível 2)
6. Quando Redis voltar, sistema repopula cache gradualmente (cache warming)

**Implementação:**
```typescript
class SaldoService {
  async getSaldoDisponivel(carteiraId: string): Promise<number> {
    try {
      const cached = await redis.get(`carteira:${carteiraId}:saldo`);
      if (cached) return parseFloat(cached);
    } catch (error) {
      logger.warn('Redis indisponível, usando fallback para DB');
    }
    
    // Fallback para DB
    const carteira = await prisma.carteirasDigitais.findUnique({
      where: { id: carteiraId }
    });
    
    return carteira.saldoDisponivel;
  }
}
```

#### Fallbacks

| Cenário | Fallback | Impacto |
|---------|----------|---------|

| **Gateway de pagamento offline** | Criar conta sem boleto, retry via job | Cliente recebe boleto por email em até 24h |

| **Message broker offline** | Persistir eventos em DB, job publica quando broker voltar | Processamento assíncrono atrasado |
