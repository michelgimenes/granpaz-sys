## 2. Contexto e Escopo

### Problem Statement
A gestão de redes de patrocínio multinível exige integridade referencial estrita para evitar fraudes financeiras (ciclos infinitos, auto-patrocínio, CPFs duplicados em árvores distintas) que resultam em pagamento indevido de bonificações. Adicionalmente, a ausência de realocações atômicas com recálculo em cascata gera inconsistências na profundidade da rede, quebrando a lógica de comissionamento por níveis. 

Sob a ótica jurídica, a justiça do trabalho brasileira possui histórico de caracterizar vínculos empregatícios em redes de venda direta quando há indícios de subordinação, cumprimento de horários ou metas punitivas. A falta de um **Air-Gap Funcional** (isolamento conceitual de termos e entidades de RH/CLT) no sistema expõe a administradora a passivos trabalhistas milionários. Este módulo estabelece o grafo hierárquico imutável, a realocação atômica e a blindagem jurídica da rede comercial.

### Objetivos
| Categoria | Métrica / KPI | Alvo |
| :--- | :--- | :--- |
| **Performance** | Latência p99 da API de consulta de árvore (`GET /rede/arvore` até 5 níveis) | `< 200ms` |
| **Integridade** | Taxa de estados intermediários (profundidade incorreta) durante realocações | `0%` (Atomicidade Estrita) |
| **Segurança** | Ocorrências de ciclos ou auto-patrocínio persistidos no DB | `0` (Bloqueio em Camada de Serviço + DB) |
| **Compliance** | Presença de campos/entidades de subordinação CLT (metas, horários) | `0` (Air-Gap Funcional) |
| **Operacional** | Tempo de exportação CSV de rede com 10.000 nós | `< 5 segundos` |

### Out of Scope (Não será implementado nesta versão)
1.  **Cálculo e Pagamento de Bonificações:** Responsabilidade do Módulo 3 (Financeiro). Este módulo apenas provê a estrutura de `nivel_profundidade`.
2.  **Cadastro Inicial do Revendedor:** Coberto pelo Módulo 1 (Contratação).
3.  **Módulos de RH/DP:** Ausência total de folha de pagamento, férias, ponto eletrônico ou avaliação de desempenho.
4.  **Split Automático de Pagamentos:** Repasses via Gateway são tratados no Módulo Financeiro.
5.  **Marketplace de Serviços:** Venda de produtos adicionais fora do escopo de patrocínio.

### Premissas e Restrições
├── **Stack:** Node.js/TypeScript, PostgreSQL 15+ (Recursos: `Recursive CTEs`, `Partial Unique Indexes`, `Advisory Locks`), Redis (Cache de Grafo).
├── **Compliance Trabalhista:** Air-Gap Funcional obrigatório. O sistema é cego para conceitos de CLT (Art. 3º da CLT).
├── **Imutabilidade de Histórico:** Registros de patrocínio nunca sofrem `DELETE` físico ou `UPDATE` de dados core. O encerramento de vínculo ocorre exclusivamente via preenchimento de `data_fim_vinculo`.
└── **Verdade Absoluta:** A profundidade (`nivel_profundidade`) é materializada na tabela para otimização de leitura, mas sua integridade é garantida transacionalmente.

### Glossário
| Termo | Definição |
| :--- | :--- |
| **Patrocínio** | Vínculo comercial unidirecional onde um revendedor indica/outro, gerando direito a bonificação por níveis. |
| **Realocação** | Mudança do patrocinador de um revendedor. Exige recálculo atômico de toda a sua sub-árvore. |
| **Cross-Tree** | Fraude onde um mesmo CPF atua ativamente como revendedor em duas árvores de patrocínio distintas simultaneamente. |
| **Air-Gap Funcional** | Isolamento arquitetural que proíbe a existência de telas, campos ou lógicas de subordinação trabalhista (metas, horários, chefia). |
| **Profundidade** | Distância (nível) de um revendedor em relação à raiz da sua árvore de patrocínio. |

### 3.2 Diagrama de Fluxo de Dados (DFD) - Realocação Atômica
```mermaid
sequenceDiagram
    participant Admin as SuperAdmin
    participant API as Network Service
    participant DB as PostgreSQL
    
    Admin->>API: POST /rede/realocar {revendedor_id, novo_patrocinador_id}
    API->>DB: BEGIN TRANSACTION
    API->>DB: 1. SELECT ... FOR UPDATE (Lock Pessimista no revendedor)
    API->>DB: 2. Valida Cross-Tree e Ciclos (CTE)
    API->>DB: 3. UPDATE patrocinios SET data_fim_vinculo = NOW() (Vínculo Antigo)
    API->>DB: 4. INSERT novo vínculo (data_fim_vinculo = NULL)
    API->>DB: 5. UPDATE em cascata (Recalcula nivel_profundidade da sub-árvore)
    API->>DB: COMMIT
    API->>API: Invalida Cache Redis da árvore afetada
    API-->>Admin: 200 OK
```

### 3.3 Architecture Decision Records (ADRs)
| ID | Decisão | Alternativas Consideradas | Justificativa | Consequências |
| :--- | :--- | :--- | :--- | :--- |
| **ADR-01** | **Recálculo em Cascata via Application Layer + CTE** | Triggers Recursivas no DB. | Triggers recursivas em PostgreSQL são propensas a loops infinitos e difíceis de debugar. Application Layer com `FOR UPDATE` e CTE é testável e previne deadlocks obscuros. | Exige transações de banco mais longas durante realocações de árvores gigantes. |
| **ADR-02** | **Histórico via Soft-Delete (`data_fim_vinculo`)** | Tabela de Auditoria separada (`patrocinios_log`). | Simplifica queries de "árvore ativa" vs "histórico" usando `Partial Unique Indexes`. Mantém a integridade referencial para bonificações passadas. | Tabelas crescem indefinidamente. Necessário particionamento por ano em fases futuras. |
| **ADR-03** | **Air-Gap Funcional (Anti-CLT)** | Incluir campos de "Gerente de Conta" ou "Meta Mensal". | 🚨 **COMPLIANCE:** Qualquer campo de subordinação gera passivo trabalhista (vínculo empregatício). O sistema deve ser estritamente comercial. | Proibição absoluta de termos como "chefe", "meta", "ponto" no código e UI. |
| **ADR-04** | **Validação Cross-Tree via Camada de Serviço** | Constraint `CHECK` complexa no DB. | Consultar árvores cruzadas via trigger é custoso e gera locks globais. Validação via Serviço com `Advisory Lock` é mais performática. | Exige que o serviço de cadastro (Módulo 1) consuma esta API antes de criar novos revendedores. |

## 4. Especificação Funcional Detalhada

### 4.1 Regras de Negócio (Determinísticas)
| ID | Regra de Negócio (SE / ENTÃO / SENÃO) |
| :--- | :--- |
| **RN-01** | 🚨 **SE** tentativa de criar vínculo onde `revendedor_id` == `patrocinador_id` OU `patrocinador_id` é dependente direto (cônjuge/filho) do `revendedor_id` **ENTÃO** bloquear com erro `AUTO_PATROCINIO_PROIBIDO`. **SENÃO** (se for parentesco indireto), permitir mas logar alerta de auditoria. |
| **RN-02** | 🔒 **SE** tentativa de cadastro/realocação de revendedor **E** o CPF já possuir vínculo ativo (`data_fim_vinculo IS NULL`) em outra árvore (Cross-Tree) **ENTÃO** bloquear com erro `CONFLITO_DE_REDE`. **SENÃO** (se o vínculo anterior estiver inativo), permitir nova alocação. |
| **RN-03** | ⚠️ **SE** solicitação de realocação de revendedor **ENTÃO** o sistema deve adquirir lock pessimista (`FOR UPDATE`), encerrar o vínculo atual (`data_fim_vinculo = NOW()`), criar novo vínculo e **recalcular atomicamente** o `nivel_profundidade` de todos os subordinados diretos e indiretos na mesma transação. **SENÃO** (se ocorrer timeout/erro), dar `ROLLBACK` integral. |
| **RN-04** | 🛡️ **SE** qualquer requisição tentar persistir campos relacionados a "cumprimento de horário", "meta punitiva", "avaliação de desempenho" ou "subordinação hierárquica funcional" **ENTÃO** a API deve rejeitar e o WAF/Code Review deve bloquear. **EXCEÇÃO:** Nenhuma. (Air-Gap Funcional). |
| **RN-05** | 💰 **SE** um revendedor for realocado para um nível que o desqualifique de uma faixa de bonificação configurada (`niveis_bonificacao`) **ENTÃO** o sistema deve recalcular os percentuais aplicáveis para vendas futuras, sem retroagir em bonificações já geradas (`transacoes_bonificacao` com status `LIBERADO`). |

### 4.4 Edge Cases e Exceções (30% da Seção - Foco Crítico)
| Cenário | Condição / Falha | Comportamento Esperado do Sistema |
| :--- | :--- | :--- |
| **EC-01: Race Condition na Realocação** | Dois admins tentam realocar o mesmo revendedor (ou sua sub-árvore) simultaneamente. | O primeiro request adquire `SELECT ... FOR UPDATE` na linha do revendedor e nas linhas dos subordinados (via CTE). O segundo request entra em *wait* no DB até o commit do primeiro, ou falha com `409 Conflict` se houver timeout de lock (ex: 5s). |
| **EC-02: Tentativa de Ciclo Indireto** | Admin tenta realocar "A" para ser patrocinado por "C", sendo que a árvore atual é A -> B -> C. | A Camada de Serviço executa uma *Recursive CTE* partindo de "C" para verificar se "A" está nos nós superiores. Se positivo, aborta com `400 Bad Request` ("CICLO_DE_PATROCINIO_DETECTADO"). |
| **EC-03: Sub-árvore Gigante (Timeout de CTE)** | Realocação de um revendedor que possui 50.000 subordinados indiretos. O recálculo estoura o timeout da API. | A API de realocação para árvores > 1.000 nós deve ser **assíncrona**. Retorna `202 Accepted`, enfileira job no BullMQ com lock distribuído e notifica o admin via WebSocket quando o recálculo em background for concluído. |
| **EC-04: Revendedor Inativo Reativado** | Revendedor teve `data_fim_vinculo` preenchido há 1 ano. Tenta-se cadastrá-lo novamente. | O sistema não cria `UPDATE` no registro antigo. Cria um **NOVO INSERT** na tabela `patrocinios` com `data_fim_vinculo = NULL` e `nivel_profundidade` recalculado. O histórico antigo permanece intacto para auditoria de bonificações passadas. |
| **EC-05: Patrocinador Inativado** | O patrocinador de "B" tem seu contrato cancelado (Módulo 4). | Job noturno identifica patrocinadores inativos e realoca automaticamente seus subordinados diretos para o "Patrocinador Raiz" (Conta da Administradora), logando o motivo como `INATIVACAO_PATROCINADOR`. |
| **EC-06: Injeção de Termos CLT via API** | Frontend manipulado envia payload com campo oculto `meta_mensal: 10000`. | O Schema de Validação (Zod/Joi) rejeita propriedades desconhecidas (`strict mode`). Retorna `400 Bad Request`. O WAF loga tentativa de injeção de vocabulário proibido. |

### 4.5 Validações de Entrada
| Campo | Regra | Mensagem de Erro | HTTP Code |
| :--- | :--- | :--- | :--- |
| `revendedor_id` | UUID válido, não pode ser igual a `patrocinador_id`. | "Revendedor não pode patrocinar a si mesmo." | 400 |
| `patrocinador_id` | UUID válido, deve possuir vínculo ativo (`data_fim_vinculo IS NULL`). | "Patrocinador alvo está inativo." | 400 |

## 5. Especificação Não-Funcional

### 5.1 Performance
├── **Latência:** API `GET /rede/arvore` (até 5 níveis) `< 200ms` (p95) utilizando cache Redis.
├── **Throughput:** Suportar 100 consultas de grafo simultâneas sem degradação do DB principal.
├── **Recálculo:** Realocação de árvores com até 500 nós deve ocorrer em `< 2 segundos`.
└── **Exportação:** CSV de 10.000 linhas gerado via *streaming* (Cursor do DB) para não estourar memória RAM (`< 5s`).

### 5.2 Escalabilidade
├── **Read Replicas:** Todas as queries de leitura de grafo (`GET /rede/arvore`, `GET /rede/subordinados`) devem ser roteadas para réplicas de leitura do PostgreSQL.
├── **Cache (Redis):** O grafo de subordinados diretos é cacheado com TTL de 10 minutos. A invalidação ocorre via eventos (`revendedor.realocado`).
└── **Particionamento:** A tabela `patrocinios` será particionada por ano (`data_entrada`) no segundo ano de operação para manter o índice parcial rápido.

### 5.3 Resiliência
├── **SLO:** `99.9%` de disponibilidade para APIs de leitura.
├── **Fallback de Grafo:** Se o DB principal estiver com latência > 1s, a API retorna o grafo "estático" do Redis (mesmo que levemente desatualizado) com header `X-Cache: STALE`.
├── **Dead Letter Queue (DLQ):** Realocações assíncronas que falham por deadlock vão para DLQ e geram alerta no PagerDuty para intervenção manual do DBA.

### 5.4 Segurança
├── **AuthN/AuthZ:**
│   ├── `REVENDEDOR`: Vê apenas sua própria sub-árvore.
│   ├── `SUPERVISOR`: Vê sua sub-árvore e métricas agregadas.
│   └── `SUPERADMIN`: Vê a floresta completa e tem permissão para realocar.
├── **Criptografia:** Dados em trânsito (TLS 1.3).
├── **Sanitização:** Motivos de realocação passam por escape de HTML para evitar XSS no Backoffice.
└── **Auditoria:** Toda realocação gera registro imutável em `audit_logs` contendo `actor_id`, `old_patrocinador_id`, `new_patrocinador_id` e `ip_address`.

### 5.5 Observabilidade
├── **Métricas (Prometheus):**
│   ├── `rede_realocacao_duracao_segundos` (Histograma)
│   ├── `rede_profundidade_media` (Gauge)
│   └── `rede_conflito_cross_tree_total` (Counter - Alerta de Fraude)
├── **Logs Estruturados:** JSON contendo `trace_id`, `revendedor_id`, `action="REALOCACAO"`, `subordinados_afetados=X`.
└── **Alertas (PagerDuty):**
    ├── Pico de `CONFLITO_DE_REDE` (> 10 em 1 hora) - Possível ataque ou bug de integração.
    └── Tempo de recálculo em cascata > 5 segundos (Risco de Timeout).

### 5.6 Acessibilidade e i18n
├── **A11y:** O componente de Grafo (Frontend) deve ter uma alternativa em formato de Lista/Árvore (DOM padrão) para leitores de tela, pois `<canvas>` ou SVGs complexos são inacessíveis.
└── **i18n:** Preparado para PT-BR e ES. Moeda e datas formatadas via `Intl`.

## 7. Estratégia de Entrega e Mitigação de Riscos

### 7.1 Plano de Rollback/Migração
├── **Migração:** Dados legados (se existirem) serão importados via script ETL que calculará a profundidade inicial e inserirá com `data_fim_vinculo = NULL`.
├── **Rollback:** Reversão do container. As constraints de DB são versionadas via Flyway. Rollback do DB envolve `DROP INDEX` e remoção de Triggers.
└── **Zero Downtime:** A API de leitura é aditiva. A API de realocação pode ser protegida por Feature Flag.

### 7.2 Feature Flags
├── `REDE_REALOCACAO_ASSINCRONA_ENABLED`: Se `true`, força realocações de árvores > 100 nós para fila (BullMQ). Se `false`, tenta processar síncrono (risco de timeout).
├── `REDE_CROSS_TREE_STRICT_MODE`: Se `true`, bloqueia CPFs que já foram revendedores inativos no passado (modo paranoia). Default: `false` (permite reativação).

### 7.3 Matriz de Riscos
| Risco | Probabilidade | Impacto | Plano de Mitigação |
| :--- | :--- | :--- | :--- |
| **Deadlock em Recálculo de Cascata** | Média | Alto | Uso de `SELECT ... FOR UPDATE` com ordenação determinística (`ORDER BY id`) para adquirir locks na mesma ordem. Timeout de transação configurado para 5s. |
| **Passivo Trabalhista (Vínculo CLT)** | Baixa | Crítico | 🛡️ **Air-Gap Funcional.** Code Review obrigatório com busca por regex (`meta`, `horario`, `chefe`, `salario`). QA valida ausência de campos na UI. |
| **Fraude Cross-Tree Não Detectada** | Baixa | Alto | Job noturno de saneamento que roda CTEs cruzando CPFs em árvores distintas para gerar relatório de auditoria. |
| **Estouro de Memória (Export CSV)** | Média | Médio | Exportação obrigatória via *Cursor* do PostgreSQL (Streaming), nunca carregando o array completo na memória do Node.js. |

### 7.4 Dependências Externas
├── **Módulo 1 (Cadastro):** Depende da API de validação Cross-Tree antes de persistir um novo `tipo_registro = REVENDEDOR`.
├── **Módulo 3 (Financeiro):** Depende da tabela `patrocinios` (via `nivel_profundidade`) para calcular bonificações multinível.
└── **Módulo 4 (Sinistro/Compliance):** Depende do status do contrato do patrocinador para disparar realocações automáticas de órfãos.

## 8. Checklist de Validação do SPEC

- [ ] **Atomicidade:** O recálculo de `nivel_profundidade` de todos os subordinados ocorre na **mesma transação** de banco da realocação do pai?
- [ ] **Air-Gap Funcional:** O Schema da API e o DOM do Frontend estão 100% livres de termos como "meta", "horário", "ponto", "chefe" ou "salário"?
- [ ] **Anti-Ciclo:** A Camada de Serviço valida ciclos indiretos (A->B->C->A) antes de persistir a realocação?
- [ ] **Cross-Tree:** O sistema impede que um CPF possua dois registros com `data_fim_vinculo IS NULL` simultaneamente?
- [ ] **Imutabilidade:** Triggers de banco impedem `DELETE` físico e `UPDATE` de campos históricos na tabela `patrocinios`?
- [ ] **Performance:** A query de leitura do grafo utiliza `Recursive CTE` com limite de profundidade (`max_nivel`) para evitar *Denial of Service* no DB?
- [ ] **Resiliência:** Realocações de árvores gigantes (> 1000 nós) possuem fallback para processamento assíncrono (Fila)?
- [ ] **Observabilidade:** Existem alertas configurados para picos de `CONFLITO_DE_REDE` e tempos de recálculo acima do SLO?
- [ ] **QA Testability:** Os cenários Gherkin cobrem a realocação em cascata e a tentativa de criação de ciclos, permitindo automação E2E?
